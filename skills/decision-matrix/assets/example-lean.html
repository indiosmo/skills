fwll: contract size on aggregates decision matrix

Group orders by an aggregating key (e.g. trader, strategy) and report a total quantity. When the underlying instruments have venue-specific contract sizes (one BTC future represents 0.01 BTC at venue A and 0.001 BTC at venue B), the aggregate is meaningless unless we normalise. Two coupled sub-decisions: how to model the size, and where in the pipeline to apply the multiplier.

Assumptions

  • Venue-specific weights: no current group uses them.

Open Questions

  • Does the contract-multiplier path work for aggregates created before the multiplier service existed? (The "?" cell in sub-decision 1.) Need a test that loads a pre-existing aggregate and verifies the multiplier resolves correctly.
  • How often does the venue change contract sizes? If less than once a year, the manual fan-out cost is negligible and the ratio column becomes more attractive.
  • Is there an audit-trail requirement on raw vs adjusted quantity? If compliance needs to reconstruct the un-adjusted order, the gateway must log both -- that nudges further toward applying the multiplier inside the gateway.
  • Will a venue ever price in non-integer multiples? The current multiplier table assumes integer multiples; revisit if a venue ships fractional contract sizes.

1. How to model the size

Criterion Add ratio column to group members Each member row carries its own multiplier; aggregate sums qty * ratio. Use contract multiplier Lookup table keyed by instrument; aggregate fetches the multiplier per row at compute time. Notes
Performance Additional lookup for every target group member when computing the aggregate. Multiplier is part of the request payload; no extra lookup. Both are O(n) over members; the ratio-column variant adds a constant per-row read. Not material at current volumes.
Instrument size changes by venue Every user must update all groups that include the affected instrument. Manual fan-out. Automatic: the venue ships the new multiplier, all groups pick it up. Venues change contract sizes more often than you would expect (quarterly splits, listing rule changes).
Flexibility (different weights on different venues) Users can configure whatever weights they want, per group. Fixed multiplier, as determined by the venue. Not supported. Would require an additional "venue" key on all lookups in the contract-multiplier path -- large added complexity.
Backward compat with existing aggregates Every existing aggregate needs a one-off backfill to gain the new ratio column. ? Not investigated. Likely fine if the multiplier table is keyed by instrument alone, but no test covers the case where an existing aggregate predates the multiplier service. The "?" goes into Open Questions below.

Decision

Use the contract multiplier. The instrument-size-changes-by-venue row decides it: a ratio column forces a manual fan-out across every aggregate any time a venue changes its contract size, and venues change contract sizes more often than expected.

The ratio column wins on flexibility (per-group venue weights), but that flexibility has not been needed in practice and can be layered on later as a per-venue override if a real use case emerges. The backward-compat gap is the only open blocker -- resolve before committing (see Open Questions).

2. Where to apply the multiplier

Criterion Pass the adjusted quantity in the request Caller multiplies before sending; gateway sees a single pre-adjusted quantity. Apply inside the gateway Caller sends raw quantity; gateway looks up the multiplier and applies it on the way through. Pass the multiplier in the request Caller sends raw quantity and the multiplier; the consumer applies it.
Clarity at the call site Hard to name in a way that conveys the purpose. Reads as a magic-number quantity. Caller code is clean, but the multiplication is hidden inside the gateway and easy to forget when debugging. Clear at the call site: quantity * contract_size is visible and named.
PnL reconstruction downstream Possible, but the multiplier has to be reversed from the adjusted quantity. Fragile if the multiplier changes mid-day. Yes -- gateway logs both raw and adjusted quantity; PnL picks whichever it needs. Yes -- both fields travel with the request.
Risk of silent PnL drift Every caller must remember to apply the multiplier. One forgotten call site means silent drift. Centralised in the gateway; one place to test and audit. Caller can pass an inconsistent pair (raw + wrong multiplier). Loud if validated, silent if not.
Development effort Touches every caller. One change in the gateway; callers untouched. Touches every caller and the consumer; also requires a schema bump.

Decision

Apply the multiplier inside the gateway. Centralising the multiplication is the only option that prevents silent PnL drift across forgotten call sites, which is the heaviest cost in the matrix.

Passing the multiplier in the request is the clearest at the call site, but the schema bump and the per-caller consistency burden aren't worth it for a transformation that is fundamentally a property of the venue, not the order. This sub-decision depends on sub-decision 1: the multiplier table must serve existing aggregates safely before the gateway can apply it -- resolve the backward-compat question first, then wire the gateway after a week of clean data.
