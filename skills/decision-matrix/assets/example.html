cpp-guides: copy vs symlink decision matrix

docs/cpp-guides/ is currently a copy of the source-of-truth at /home/msi/llm_workspace/skills/docs/cpp-guides/. The question is whether to keep copying, switch to a symlink, or use a git-native sharing mechanism. Two adjacent options are included for honest comparison.

A. Copy (current)

Plain files committed in docs/cpp-guides/. Periodic manual resync from origin.

B. Symlink

docs/cpp-guides → /home/msi/llm_workspace/skills/docs/cpp-guides. Symlink itself is git-tracked.

C. Git submodule

Origin is its own repo; this project pins a commit SHA. git submodule update to refresh.

D. Git subtree

Origin's history merged in; files live here as normal commits. Pull updates with git subtree pull.

Interpretation

Symlink is the weakest option despite winning on drift and single-source. The two killers are machine portability (any CI, any other contributor, any container sees a dangling link) and blast radius (every edit silently leaks into every project sharing the link, eliminating the ability to try changes locally). The weighted score confirms what the cell pattern already shows.

Copy and subtree are the strong options, with subtree slightly ahead on the scoreboard. Copy is what's there; it's fine if "resync from origin" stays a manual chore -- the guides are short enough that drift is visible during normal review. Subtree is the upgrade path if drift becomes painful: same local-files ergonomics as copy, plus a real command (git subtree pull --prefix=docs/cpp-guides ...) instead of cp -r, and an explicit history of when origin was last pulled in.

Submodule is overkill unless the guides start versioning independently and other consumers need to pin specific versions. The two-step commit dance and the clone-time footgun outweigh the version-pinning benefit for a docs tree this small.

Leaning toward: stay on copy for now; revisit subtree the first time drift bites (e.g. running cp -r more than once a month, or shipping a doc fix here and forgetting to port it back). If origin isn't yet its own git repo, that's the prerequisite for moving to subtree.

Open Questions

  • Is the origin already its own git repo? If /home/msi/llm_workspace/skills/ is one big repo, subtree needs a specific path or a split-out first.
  • How many other projects consume cpp-guides? Symlink's blast-radius cost is proportional to that number. If this is the only consumer today, symlink's downside is theoretical -- but path portability still kills it.
  • How often does cpp-guides actually change? If it's "once a quarter", copy is genuinely fine and the whole conversation is premature. If it's weekly, drift will hurt.
  • Is there ever a project-specific edit to a guide (e.g. an abacus-specific example) that should not pollute the generic doc? Copy and subtree allow it; symlink forbids it; submodule requires a branch.
