- Avoid glyphs/icons in general, both in comments and in cli/log outputs. Just use regular text.
- Variable names should respect the domain. Use the vocabulary practitioners use, rather than generic substitutes: in algorithmic trading code, prefer names like `bid`, `ask`, `resting_price`, `top_of_book`, `aggressor`, `liquidity`, `taker`, `maker`, and `rebate` when those concepts apply. In clinical medicine code, prefer names like `encounter`, `triage`, `diagnosis`, `contraindication`, `dosage`, `specimen`, and `lab_result`. In logistics code, prefer names like `shipment`, `consignment`, `carrier`, `waybill`, `fulfillment_center`, `pick_list`, and `backorder`. In digital audio code, prefer names like `sample_rate`, `buffer_size`, `latency`, `gain`, `clipping`, `noise_floor`, and `wet_signal`. When the domain language is unclear, research the jargon or ask the user before naming core concepts.
- Never abbreviate variable, function, type, file, or field names merely to make them shorter. Avoid names like `pb_file`, `inv`, `env_resp`, `patient_dx`, `ship_dt`, `buf_sz`, and `auth_req`; use `playbook_file`, `inventory`, `environment_response`, `patient_diagnosis`, `shipment_date`, `buffer_size`, and `authorization_request`. Use an abbreviation only when it is well accepted in the domain and the name is localized in scope, such as `sku` in commerce or logistics, `mrn` in clinical systems, `http` in web code, or `cpu` in systems code.
- Comments and docs describe what the code in front of the reader does, and why if the why is non-obvious. They do not describe what the code used to do, what it no longer does, what lives elsewhere, or what the caller is expected to handle. A reader seeing the code for the first time has no context for absent behavior, and enumerating non-responsibilities only makes sense as a contrast with a previous version they cannot see. The code shows what it does; git history shows what it used to do. If you cannot phrase a comment as a positive statement about the present code, delete it. This is the "negative documentation" anti-pattern from the documentation skill, and it shows up most often during refactors.

  Watch for these tells -- any of them means the comment is describing absent behavior:
    - past-tense framing: "we used to ...", "previously ...", "this no longer ..."
    - delta framing: "now handled upstream", "dedup moved to ...", "X is no longer joined here"
    - non-responsibility framing: "X is the caller's responsibility", "Y is handled elsewhere", "this does not validate Z", "integration of the result is up to the caller"

  Examples:

    BAD (past-tense): `// we used to dedupe here, but the upstream stage now guarantees uniqueness`
    GOOD: delete the comment. The function dedupes or it doesn't; the code shows which.

    BAD (delta): `// status resolution moved to the runtime layer`
    GOOD: delete the comment. If the runtime layer's role needs explaining, document it where that layer lives.

    BAD (non-responsibility): `/* Stateless converters ... integration of the resulting events (e.g. routing, status resolution) is the caller's responsibility. */`
    GOOD: `/* Stateless converters from OnixS B3 UMDF messages to macuco market data domain types. */` -- describe what the functions do; do not enumerate what they don't do.

    BAD (non-responsibility, inline): `// caller must hold the session mutex`
    GOOD: if the locking contract is non-obvious and load-bearing, that is a legitimate "why" -- keep it, but phrase it as a precondition the function relies on, not a chore the caller has been assigned: `// precondition: session mutex is held; we read session_.next_seq without locking`

- Treat `work-in-progress/` as ephemeral feature-work space. Files there may be created while a feature is in flight and deleted when the work is done, so keep references to `work-in-progress/` out of code, permanent documentation, configuration, and any other durable project artifact.
- When multiple viable approaches exist for solving a problem, build a decision matrix to guide the choice. Identify the candidate options and the criteria that matter (e.g. complexity, performance, reversibility, blast radius, maintenance cost, ergonomics, testability, coupling, etc), and score each option against the criteria. Present the matrix and the options for the user to decide, unless there is a very clear winner.
- You are running inside a sandbox. Both Claude Code and Codex CLI are configured to run every shell command under OS-level sandboxing (bubblewrap on Linux/WSL2, Seatbelt on macOS for Claude; the equivalent in Codex), with managed settings that the local agent cannot loosen. There is no escape hatch: Claude has `allowUnsandboxedCommands: false`, so `dangerouslyDisableSandbox` is silently ignored; Codex permits only `read-only` and `workspace-write` modes, never `danger-full-access`. Do not try to work around the sandbox -- work with it. The constraints are stable within a session; when a command is rejected, read the rejection and adjust, rather than retrying variations until something sticks.
- The most common friction points are: (1) writes outside the project root or `$TMPDIR` (use `$TMPDIR`, never `/tmp` directly, for scratch files); (2) reads under `~/.ssh`, `~/.aws`, `~/.config/gh`; (3) network calls from `bash` to hosts that are not on the allowlist; (4) git write operations (`push`, destructive resets, `--no-verify`), which a managed `PreToolUse` hook blocks; (5) edits to `.claude/settings.json`, `/etc/claude-code/managed-settings.*`, or the project's `config/` directory. If you hit a sandbox wall and the path forward is not obvious, consult `claude_sandbox.md` or `codex_sandbox.md` (whichever matches the CLI you are running under) for the full rules and the recommended workarounds.
- Three-strikes rule. If a command keeps getting refused or keeps failing in a way you suspect is sandbox-related, stop after the third attempt. Do not keep iterating variations. Either ask the user to run the command themselves, or tell them what you were trying to do and ask how they would like to proceed. Repeated refusals within a session almost never resolve on the fourth try -- the policy is stable, and continued attempts just burn context.
- Distinguish sandbox refusals from shell-quoting failures. A sandbox refusal looks like "Operation not permitted", "Permission denied", or an explicit allowlist-related message naming the path or host. A shell-quoting failure looks like variables appearing literally in output (`$!`, `$BGPID`), `process ID list syntax error` from `ps`, or commands behaving as if a variable was empty. These are not sandbox issues -- bubblewrap and Seatbelt isolate filesystem and network, they do not change how bash parses or expands variables. For non-trivial compound commands with background processes, here-docs, or variable interpolation across statements, write a script to `$TMPDIR` with the `Write` tool and run it with `bash "$TMPDIR/script.sh"` instead of inlining the whole thing in a single `Bash` call.
